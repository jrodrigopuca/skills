# AGENTS.md Generator

Generate accurate, evidence-based `AGENTS.md` files following the [agents.md](https://agents.md/) open standard.

## Overview

**README.md is for humans. AGENTS.md is for agents.** It gives AI coding agents the context they need to work in a repository: setup commands, how to build and test, code conventions, and project-specific gotchas — without cluttering the human-facing docs.

This skill generates AGENTS.md from **repository evidence**, not from assumptions. Every command it documents must exist in the repo; every convention it states must be observable in configs or code.

## Core Principles

1. **Evidence over invention.** Only document commands found in `package.json` scripts, Makefiles, CI workflows, or task runners. Only state conventions enforced by configs (linter, formatter, tsconfig) or clearly observable in the code.
2. **Token economy.** Agents read this file on nearly every session — every line costs context, forever. Target **under 150 lines**. If a topic needs depth, link to a doc instead of inlining it.
3. **Instructions, not descriptions.** "Run `pnpm test:unit` before committing" beats three paragraphs about the testing philosophy. Write imperatives an agent can act on.
4. **Don't duplicate the README.** Link to it for background. AGENTS.md holds only what an agent needs that the README doesn't say (or says too verbosely).
5. **Closest file wins.** In monorepos, nested `AGENTS.md` files take precedence over the root one for files in their subtree — use this to keep each file small and specific.

## Instructions

### 1. Gather Evidence

Inspect the repository before writing anything:

| Evidence source                                     | What it tells you                        |
| --------------------------------------------------- | ---------------------------------------- |
| `package.json` scripts / `Makefile` / `justfile`    | Real build, test, lint, dev commands     |
| Lockfile (`pnpm-lock.yaml`, `yarn.lock`, `uv.lock`) | Package manager to use (never guess)     |
| CI workflows (`.github/workflows/*`)                | The commands that MUST pass; Node/Python versions |
| Linter/formatter configs (eslint, prettier, ruff)   | Enforced code style — cite, don't restate |
| `tsconfig.json`, `pyproject.toml`                   | Strictness, paths, target versions       |
| Test configs (jest/vitest/playwright/pytest)        | Test layout, how to run a single test    |
| `.env.example`, docker-compose                      | Required environment for local runs      |
| Existing `CONTRIBUTING.md`, `docs/`                 | Conventions already written down — link  |
| Git history (`git log --oneline -20`)               | Commit message convention actually used  |

If the project already has documentation generated by `create-software-docs` (a `docs/` directory with development-guide.md), reuse its findings and **link to those docs** instead of repeating them.

### 2. Determine the Mode

- **generate** — no AGENTS.md exists: create it from evidence.
- **update** — AGENTS.md exists: preserve human-written rules, fix stale commands, add missing sections. Never silently delete a rule you didn't write; if evidence contradicts it, flag it to the user.

### 3. Choose the Scope (Monorepo Rule)

- **Single project** → one `AGENTS.md` at the repo root.
- **Monorepo** → a short root `AGENTS.md` (workspace-level commands, cross-cutting conventions) plus one per package/app that differs meaningfully (its own commands, framework, or style). Agents follow the **closest file to the edited file**, so nested files only need the deltas — don't repeat the root content.

### 4. Write the Sections

Use this structure, omitting sections with nothing evidence-backed to say:

```markdown
# AGENTS.md

## Project overview
{2-3 lines: what this is, main stack, where the code lives}

## Setup
{install command with the CORRECT package manager, env requirements}

## Commands
{build / test / lint / dev — verified commands only, including how
to run a SINGLE test, which agents need constantly}

## Code style
{only what configs enforce or the codebase consistently shows:
"TypeScript strict, no default exports, Prettier via `npm run format`"}

## Testing instructions
{where tests live, naming pattern, what must pass before a commit/PR}

## Commit & PR conventions
{observed commit format — e.g. Conventional Commits — and PR expectations}

## Security & boundaries
{files agents must not touch, secrets handling, generated code, migrations policy}

## Gotchas
{the non-obvious things that waste agent time: "the dev server needs
docker compose up first", "snapshots are regenerated with test:update"}
```

See [references/templates.md](references/templates.md) for filled-in templates per project type (app, library, Python service, monorepo root + package).

### 5. Verify Before Delivering

Run the commands you documented (or at minimum verify they exist as script entries):

- Install/setup command matches the lockfile's package manager
- Build, lint, and test commands exist and are spelled exactly right
- The "run a single test" incantation is correct for the framework

**A wrong command in AGENTS.md is worse than no AGENTS.md** — agents will trust it and burn time on failures.

### 6. Wire Up Tool Compatibility

AGENTS.md is the cross-tool standard, but some tools read their own file. Ask the user which tools their team uses and apply [references/compatibility.md](references/compatibility.md). The common case:

```bash
# Claude Code reads CLAUDE.md — point it at AGENTS.md
ln -s AGENTS.md CLAUDE.md      # symlink, or:
echo "@AGENTS.md" > CLAUDE.md  # CLAUDE.md import syntax
```

## Anti-Patterns

- ❌ **The README dump** — copying the whole README in. Link it.
- ❌ **Generic advice** — "write clean code", "follow best practices". Zero information; delete.
- ❌ **The file inventory** — listing every directory. Agents can list files themselves; only explain structure that is surprising.
- ❌ **Unverified commands** — documenting `npm run test:e2e` because it "should exist".
- ❌ **The 400-line AGENTS.md** — that context tax is paid on every agent session. Split into nested files or link out.
- ❌ **Restating linter rules** — the linter enforces them; write "lint must pass: `npm run lint`" and move on.

## Example

**User:** "Create an AGENTS.md for this repo"

**Evidence found:** `pnpm-lock.yaml`; scripts: `dev`, `build`, `test` (vitest), `lint`; CI runs `pnpm lint && pnpm test`; strict tsconfig; commits follow Conventional Commits.

**Output (excerpt):**

```markdown
# AGENTS.md

## Project overview
SaaS dashboard — React 19 + Vite + TypeScript (strict). App code in `src/`, server in `server/`.

## Setup
pnpm install        # pnpm only — lockfile is pnpm-lock.yaml

## Commands
pnpm dev            # local dev server on :5173
pnpm build          # production build (must pass before PR)
pnpm test           # vitest run
pnpm vitest run src/utils/date.test.ts   # single file
pnpm lint           # eslint — CI fails if this fails

## Commit & PR conventions
Conventional Commits (feat/fix/refactor + scope). CI requires lint + tests green.

## Gotchas
- `pnpm dev` needs `docker compose up -d` first (postgres).
- Never edit `src/generated/**` — regenerate with `pnpm codegen`.
```

## When NOT to Use

- The repo already has a good AGENTS.md and nothing changed (don't churn it)
- Full project documentation is what's needed → use `create-software-docs`
- Component-level docs → use `create-component-docs`
